Executive Summary

What The Studio Is Right Now

Program Kanban is not the main conversational brain. It is the operator-facing control room around a governed multi-agent Python runtime. Chat or Codex is used for intent, steering, clarification, and final judgment. The framework handles classification, task packets, routing, approvals, evidence capture, Kanban transitions, and bounded API execution.

What it is

  • A governed control plane around project work
  • A persistent evidence system backed by SQLite
  • A tiered API execution fabric for bounded downstream work
  • An operator wall for approvals, run inspection, restore, and trace visibility

What it is not

  • Not a second free-form Orchestrator chat client
  • Not a rootless collection of prompts and loose scripts
  • Not a system where tasks can mutate state outside the board model
  • Not a repo where historical duplicate folders should be trusted equally
Working principle: the operator thinks in chat, the control room proves what happened, and the runtime only delegates work through bounded, reviewable, and persisted execution paths.

Vision And Evolution

How The Project Evolved

The repo started as a manual markdown-driven studio and evolved into a more durable multi-agent framework with explicit authority boundaries, task packets, approvals, traces, and a control-room UI. The current architecture is the result of that evolution, not a blank-slate greenfield design.

Manual studio origins

The earliest workflow leaned on markdown memory, manual handoffs, and conversational continuity more than on durable runtime state.

Repository normalization

The repo was rebuilt around stable folders for governance/, agents/, sessions/, projects/, scripts/, skills/, and tests/.

M2 operator-client trust boundary

The project explicitly chose chat as the best orchestration surface and Program Kanban as the trusted evidence surface for approvals, trace visibility, and control-room operations.

M5 SDK migration target

The repo introduced the OpenAI Agents SDK dependency and bridge concepts, but kept the control room, SQLite store, and safety model as the non-negotiable framework foundation.

M6 trust, restore, and operator speed

The current emphasis shifted from "can the studio orchestrate at all?" to "can the operator trust it, inspect it quickly, and recover safely from bad turns?"

2026-04 API-first architecture baseline

The active design direction formalized a hybrid model: deterministic local control plane plus governed OpenAI API execution for bounded planning and implementation work.

Important reading rule: milestone specs explain why the studio looks the way it does, but they are not the same thing as live code. Use them for intent, then confirm behavior in implementation files and tests.

Process And Workflow

End-To-End Operating Flow

The current process is intentionally bounded. Work does not move from operator request to specialist output by free-form delegation; it moves through classification, compilation, authorization, storage, and review.

1. Operator entry

The request enters through chat/Codex, the CLI, or the control-room web wrapper. Primary operational entrypoints are scripts/cli.py and scripts/operator_api.py.

2. Intent gateway

intake/gateway.py classifies the request as ROUTE_TASK, ROUTE_STATUS, ROUTE_ADMIN, NEEDS_SPLIT, or REJECT. Mixed or unsafe requests fail closed.

3. Task packet compilation

intake/compiler.py creates a TaskPacket with normalized request text, allowed roles, allowed tools, forbidden actions, and token budget limits.

4. Orchestrator binding

agents/orchestrator.py loads the authoritative root, attaches the task packet to team state, creates tasks/runs in sessions/studio.db, and ensures only routable packets proceed.

5. PM decomposition

agents/pm.py breaks parent tasks into deliverable-backed subtasks using the project capability registry in agents/config.py. Each subtask inherits budget, route, lane, and tool restrictions.

6. Worker manifest sealing

scripts/worker.py and skills/tools.py require an exact-path write manifest. Delegated workers may only write the approved artifact path and only with approved tools.

7. Governed API execution

agents/api_router.py validates authority packets and budget reservations, then routes the call through wrappers/llm_wrapper.py into runtime.py, which uses the OpenAI Responses API.

8. Evidence capture

The router, store, and logging layers persist usage events, trace events, validation results, artifacts, governed external call records, and reconciliation state into SQLite and markdown logs.

9. QA and board review

The QA role reviews subtask artifacts, the board enforces legal state transitions, and the final move to Done requires review consensus.

10. Operator inspection and recovery

The Program Kanban UI reads snapshots, approvals, run details, trust worklists, and restore controls through Python wrappers rather than by mutating the database directly from the browser.

Boundary to preserve: direct board-control actions remain deterministic framework code paths. The system explicitly avoids creating a second uncontrolled "agent manager" layer that can bypass storage, governance, or review.

Task Governance, Naming, And Milestones

How Tasks Are Created, Grouped, And Controlled

The wiki previously described high-level flow but did not capture the operating discipline that now governs task creation, naming, milestone grouping, and readiness. The authoritative implementation lives in sessions/store.py, governance/TASK_DECOMPOSITION.md, and the rendered board view in projects/program-kanban/execution/KANBAN.md.

Core rule: no delegated task should go directly to execution without decomposition. Deterministic local board-control actions are the only intentional exception.

Canonical task IDs

  • program-kanban tasks use the PK-### prefix.
  • tactics-game tasks use the TG-### prefix.
  • TGD-### is still accepted as a legacy alias for program-kanban parsing, but it should not be used for new work.
  • The board lifecycle is backlog, ready, in_progress, in_review, completed, blocked.

Milestone naming convention

  • Milestones are named as M<number> - <Capability Wave Name>.
  • Examples already in use include M8 - Visual Production And Digital Asset Management, M9 - Audio Production Pipeline, and M10 - Platform Scaling And Operational Follow-Ons.
  • Milestones should represent a delivery wave or capability theme, not a temporary task bucket.
  • Each milestone carries an explicit entry_goal, exit_goal, owner_role, and ordered placement.

Exact output paths

  • Every governed task should declare a single canonical expected_artifact_path.
  • Use repo-relative paths such as projects/program-kanban/app/app.js, sessions/store.py, scripts/worker.py, or projects/program-kanban/governance/LARGE_MEDIA_POLICY.md.
  • That path is reused by PM planning, manifest sealing, worker write controls, and board rendering.
  • Another LLM should treat the approved artifact path as the write boundary, not as a suggestion.

Task creation minimum standard

A Program Kanban task should not be considered properly formed unless it includes the following fields either directly on the task row or in acceptance metadata:

  • title and details describing the work without relying on chat history
  • objective describing the intended outcome
  • owner_role and usually assigned_role identifying who is responsible
  • expected_artifact_path pointing to the exact governed output location
  • acceptance metadata carrying milestone intent, approved decisions, and task constraints
  • milestone_id or enough milestone metadata for auto-resolution

Milestone auto-resolution contract

Program Kanban tasks can propose milestone creation through acceptance metadata. The store resolves and creates milestones from these fields when needed:

{
  "proposed_milestone": "M8 - Visual Production And Digital Asset Management",
  "entry_goal": "Visual review is still conflated with specialist behavior.",
  "exit_goal": "The deterministic media-service boundary is operator-visible.",
  "milestone_status": "planned",
  "milestone_owner_role": "Project Orchestrator",
  "milestone_details": "Optional longer explanation",
  "milestone_slug": "m8-visual-production"
}

If milestone_order is not supplied when a milestone is created, the store assigns the next value after the current project maximum. Milestones are listed in milestone_order ASC, created_at ASC.

Control Area Implemented Rule Practical Meaning
Decomposition governance/TASK_DECOMPOSITION.md requires classification, tier assignment, and decomposition before delegated execution. Parent tasks are profiled first, then PM converts them into bounded subtasks instead of sending vague requests directly to workers.
Ready gate For program-kanban, moving to ready requires milestone assignment, acceptance criteria, expected artifact path, and a responsible owner or assigned role. A task is not truly buildable until scope, owner, deliverable, and milestone are explicit in durable state.
Completed gate Moving to completed requires the same readiness fields plus review_state = Accepted. Finished code or docs are not enough by themselves; completion is review-backed closure.
Subtask inheritance create_subtask() inherits milestone_id from the parent task. Milestone grouping stays coherent even when work is decomposed across specialists.
PM propagation PM-generated subtasks persist deliverable_type, allowed_tools, deliverable_contract, assigned_tier, and expected_output_format. Downstream specialists receive a concrete operating contract instead of inferring format, write scope, or tool authority.

How milestone grouping should be done

  • Group tasks by major delivery objective or operating capability wave, not by who is currently working on them.
  • Prefer extending an existing milestone when a task clearly advances that same capability boundary.
  • Create a new milestone only when the entry and exit conditions define a distinct wave of value or governance.
  • Keep milestone titles stable once used in tasks and artifacts; rename only through an intentional migration, not ad hoc edits.
  • Use milestone order to express roadmap sequence. Do not infer order from creation date alone.

How another LLM should create new tasks

  • Choose the correct project prefix first: PK for Program Kanban, TG for Tactics Game.
  • Write the task so it can stand alone without conversational memory.
  • Attach or resolve the milestone before attempting to mark the task ready.
  • Set an exact governed artifact path and keep the task scoped to that output.
  • When decomposing, preserve the parent's milestone and pass explicit tool and deliverable contracts into the subtask acceptance metadata.
  • Do not mark a Program Kanban task completed until review has explicitly set review_state to Accepted.

API Use And Token Control

How External Model Calls Are Managed

What is implemented now

  • runtime.py uses client.responses.create() with store=True.
  • agents/api_router.py resolves tier, model, reasoning effort, lane, and route family.
  • intake/models.py task packets enforce max_prompt_tokens, max_completion_tokens, max_total_tokens, and max_retries.
  • agents/cost_tracker.py records input, cached input, output, and reasoning tokens and estimates cost.
  • Governed external observability records proof, claim, trust, retry, and reconciliation data.
  • Background execution is supported through the Responses API background flag and route lane metadata.

What is documented but not fully live

  • Prompt-cache shaping is defined as policy, but the current runtime does not pass cache-key parameters.
  • Batch execution is part of the architecture direction, but is not the obvious live default in the current router implementation.
  • The SDK runtime migration is scaffolded but still blocked from becoming the governed default path.
Current active API call shape (simplified):

1. Orchestrator / PM / worker prepares bounded task context
2. llm_wrapper enforces:
   - allowed role
   - allowed board state
   - forbidden prompt patterns
   - allowed tool access
3. api_router validates:
   - authority packet fields
   - budget reservation
   - retry rules
   - execution lane
4. runtime.py sends:
   - model
   - input messages
   - reasoning effort
   - metadata
   - background flag
   - store=True
5. cost_tracker and store persist:
   - prompt/output/reasoning tokens
   - cached input tokens
   - estimated cost
   - trace / proof / reconciliation metadata
Token optimization summary: the repo already enforces token budgets and logs cached token usage when returned by the provider. The larger cache-optimization strategy exists mostly as architecture policy and should not be mistaken for already-wired runtime behavior.

Error Handling And Recovery

How The System Fails Closed And Recovers

Fail-closed behavior

  • Unsafe or mixed requests are rejected or forced into NEEDS_SPLIT at intake.
  • Invalid roles, states, tools, and forbidden prompt content raise exceptions before provider invocation.
  • Governed worker execution without a valid manifest is rejected.
  • SDK mode is blocked for governed specialist execution until evidence parity exists.

API retry behavior

  • agents/api_router.py retries retryable provider failures.
  • The retry set includes rate limits, timeouts, connection errors, and internal server errors.
  • Retry backoff is simple and explicit: time.sleep(0.5 * (attempt + 1)).
  • Budget authority can stop execution even if the provider would otherwise succeed.

Recovery surfaces

  • sessions/backups/ stores backup assets and restore receipts.
  • scripts/control_room_api.py exposes restore-backup operations to the control room.
  • sessions/runtime_logs/ stores process stdout/stderr logs for launcher troubleshooting.
  • The operator UI surfaces backup and launcher status instead of requiring raw SQL or manual file juggling.
Recovery rule: recover from backups through store-backed code paths and the control-room wrapper, not by manually editing SQLite files in place.

Reuse Guide For Another LLM

How To Safely Reuse This Codebase

Safe starting order

  1. Verify the authoritative root in .workspace_authority.json.
  2. Read workspace_root.py so you do not operate in the duplicate workspace.
  3. Read governance/CONTROL_INVARIANTS.md and governance/rules.yml.
  4. Read intake/models.py, intake/gateway.py, and intake/compiler.py.
  5. Read agents/orchestrator.py, agents/pm.py, and scripts/worker.py.
  6. Read wrappers/llm_wrapper.py, agents/api_router.py, and sessions/store.py.
  7. Only then move into project UI or artifact-specific code.

Files most worth reusing

  • workspace_root.py for authority enforcement
  • intake/models.py for structured task packets and budget contracts
  • agents/api_router.py for governed tiered Responses API execution
  • skills/tools.py for exact-path write manifests
  • sessions/store.py for canonical persistence and audit patterns
  • projects/program-kanban/app/server.mjs for a thin Node wrapper around Python services

Things to ignore or treat cautiously

  • projects/tactics-game/app/node_modules/ and other generated bundles
  • Old root paths embedded in imported briefs and older specs
  • The duplicate Documentos repo copy
  • Assuming prompt caching or batch execution is already wired just because policy docs mention them
  • Assuming the SDK path is the active default because the dependency is installed
