# Reset Start Here

This folder is the reset bootstrap bundle for a brand new Codex working in a brand new clean repo.

Use this folder as the one-source authority for the reset.
Do not treat the old repo body, its legacy milestone chain, or its markdown backlog history as the active rewrite target.

## 1. What The New Repo Is Trying To Build
- Build `AIOffice` as an admin-only, self-build-first product for governed software production.
- Preserve the core product idea:
  - a thin outer control harness around untrusted models
  - operator direction above control-kernel authority above bounded executor output
  - artifact-backed, reviewable work rather than narration-backed workflow theater
- Preserve the intended product direction without over-claiming present implementation:
  - API-first and cost-governed operation
  - controlled execution profiles and explicit escalation by cost or ambiguity
  - Git and persisted state as truth substrates
  - product ambition beyond a pure governance kernel
- Broad UI or control-room work is deferred.

## 2. First Acceptable Proof Boundary
- The first acceptable proof boundary in the new repo is:
  - narrow supervised operation through `architect`
  - plus bounded `apply` or promotion control
- Stop there for the first proof slice.
- Do not claim live later-stage workflow, unattended operation, or a broad operator workspace.

## 3. Authoritative Files In This Bundle
- `RESET_START_HERE.md`
  - bootstrap instructions and order of work
- `REBASELINE_DECISION.md`
  - frozen reset decisions for the new repo
- `REBASELINE_SOURCE_MAP.md`
  - what to keep as signal, what to rewrite, what to archive or reference, and what not to carry
- `PRESERVED_PATTERNS.md`
  - the enduring doctrine, authority, workflow, truth, and process patterns worth preserving
- `COPY_MANIFEST.md`
  - the selective donor import manifest
- `OLD_REPO_FREEZE.md`
  - informational freeze note for the old repo; this is reference context, not the new product baseline

## 4. What To Ignore From The Old Repo
- Ignore old milestone and task history as active planning input.
- Ignore old `KANBAN.md`, `ACTIVE_STATE.md`, and milestone closeout chain as the starting backlog for the new repo.
- Ignore any implication that the broader product baseline is already implemented.
- Ignore broad UI or control-room ambition as an early implementation requirement.
- Ignore donor noise that came from preserving historical continuity instead of building a clean first proof.

## 5. Order For The New Codex
1. Read this file.
2. Read `REBASELINE_DECISION.md` and treat it as binding for the reset.
3. Read `PRESERVED_PATTERNS.md` and carry those patterns forward explicitly.
4. Use `REBASELINE_SOURCE_MAP.md` and `COPY_MANIFEST.md` to decide what to copy, adapt, reference, or discard.
5. Create fresh constitutional docs in the new repo:
   - `VISION.md`
   - `PROJECT.md`
   - `KANBAN.md`
   - `ACTIVE_STATE.md`
   - any minimal control-kernel and workflow docs needed for the first proof slice
6. Rebuild the first proof slice cleanly:
   - doctrine first
   - truth substrates first
   - bounded workflow through `architect`
   - bounded `apply` or promotion control
7. Only after the new clean baseline exists should donor code or donor docs be consulted selectively.

## 6. Reset Rules
- Do not migrate legacy tasks, milestones, or kanban history.
- Do not preserve noise just because it already exists.
- Preserve doctrine, naming discipline, useful governance and control patterns, and proven process rules.
- Keep the new repo honest about proof boundaries at every step.
