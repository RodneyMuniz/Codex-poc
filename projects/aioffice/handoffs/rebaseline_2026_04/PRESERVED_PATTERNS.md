# Preserved Patterns

This file captures only the enduring patterns worth preserving into the new clean repo.

## 1. Doctrine
- Fail closed over fail open.
- Evidence outranks narration.
- Artifacts outrank summaries.
- Explicit approval outranks implied permission.
- One authoritative truth surface per concern.
- Current-proof honesty outranks product ambition.

## 2. Authority Pattern
- Authority order:
  1. operator
  2. control kernel
  3. reasoning or orchestration assistance
  4. bounded executor
- Codex or any executor may produce bounded work but must not become the control plane.
- Chat may carry intent but must not become silent authority.

## 3. Workflow Control Pattern
- Use explicit packets and bounded output instead of free-form executor authority.
- Preserve a packet-out / bundle-back / review-before-mutation flow.
- Separate:
  - intent capture
  - structured collaboration
  - bounded execution
  - evidence return
  - review or apply or reject
  - durable state update
- Stop safely when proof is insufficient.

## 4. Truth And Proof Pattern
- Git commits, branches, tags, and external review surfaces matter.
- Persisted state matters.
- Local drafts and model narration are non-authoritative until reconciled against sanctioned truth surfaces.
- Keep local state, remote publication state, and accepted truth separate in reporting.

## 5. Product Framing Pattern
- AIOffice is a product for governed software production, not just a governance kernel.
- The harness is the product:
  - cost-governed
  - API-first
  - execution-profile aware
  - explicit about what is target-state only versus currently proven
- Broad UI or control-room surfaces are downstream, not first-proof requirements.

## 6. Naming And Process Patterns
- Use clear stage names:
  - `intake`
  - `pm`
  - `context_audit`
  - `architect`
  - later lanes only when proved
- Use explicit artifact names and contract names instead of role-play labels.
- Preserve `v1` contract naming where it helps keep first-proof scope narrow.
- Use explicit `proved` and `did not prove` sections in review artifacts.
- Use explicit non-claims whenever product breadth exceeds current proof.

## 7. Reset Interpretation
- Preserve patterns, not baggage.
- Preserve control law, not milestone drift.
- Preserve good naming and reporting discipline, not the old backlog history.
